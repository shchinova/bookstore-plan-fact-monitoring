# Описание исходных данных

> Документ описывает исходные данные, которые генерируются с помощью `scripts/generate_data.py`: структуру данных, описание таблиц и полей, описание намеренных аномалий и рекомендации по работе с сырыми данными.

### 1. Общая структура и режимы работы

Скрипт `generate_data.py` создаёт набор CSV-файлов в директории `data/raw/`, имитирующих реальные данные для план-фактного анализа книжного онлайн-магазина.

**Интерактивный выбор режима:**
- При запуске запрашивается: `Генерировать с аномалиями? (y/n)`
- `anomalies_enabled = True` – в данные вносятся намеренные искажения (описаны ниже).
- `anomalies_enabled = False` – генерируются «чистые» данные без аномалий.

**Воспроизводимость:** фиксированный seed = 42 для всех генераторов случайных чисел.

**Выходные таблицы:**

| Таблица          | Тип            | Разбиение по дате                        |
| ---------------- | -------------- | ---------------------------------------- |
| `dim_date`       | Справочник     | нет                                      |
| `dim_product`    | Справочник     | нет                                      |
| `fact_plan`      | План           | нет                                      |
| `fact_sales`     | Факты продаж   | `sales_history` / `sales_update`         |
| `fact_inventory` | Факты остатков | `inventory_history` / `inventory_update` |

### 2. Описание создаваемых таблиц и полей

#### 2.1. `dim_date` (Календарь)

| Поле            | Тип в Pandas              | Предполагаемый тип в БД | Описание                                    |
| --------------- | ------------------------- | ----------------------- | ------------------------------------------- |
| `date`          | `datetime64[ns]` → `date` | `DATE`                  | Дата (2024-01-01 - 2026-04-15)              |
| `year`          | `int64`                   | `INTEGER`               | Год                                         |
| `month`         | `int64`                   | `INTEGER`               | Месяц (1-12)                                |
| `quarter`       | `int64`                   | `INTEGER`               | Квартал (1-4)                               |
| `is_weekend`    | `int64`                   | `INTEGER`               | Признак выходного (0/1)                     |
| `is_holiday_ru` | `int64`                   | `INTEGER`               | Признак государственного праздника РФ (0/1) |

**Особенности:** праздники заданы жёстким списком дат (январские каникулы, 23 февраля, 8 марта, 1 и 9 мая).

#### 2.2. `dim_product` (Справочник продуктов)

Содержит ~952 записи (книги + 2 подписки).

| Поле              | Тип Pandas            | Предполагаемый тип в БД | Описание                                                                      |
|-------------------|-----------------------|-------------------------|-------------------------------------------------------------------------------|
| `product_id`      | `int64`               | `INTEGER` / `SERIAL` PK | Уникальный идентификатор товара                                                |
| `isbn`            | `object` (строка)     | `VARCHAR(13)`           | ISBN-13 (может быть `None` для подписок)                                      |
| `title`           | `object`              | `TEXT`                  | Название книги                                                                |
| `author`          | `object`              | `TEXT`                  | Автор (может быть `None` для подписок)                                        |
| `publisher`       | `object`              | `TEXT`                  | Издательство (может быть `None`)                                              |
| `published_date`  | `object` (дата)       | `DATE`                  | Дата публикации (может быть `None`)                                           |
| `language`        | `object`              | `VARCHAR(20)`           | Язык (`'Russian'`, `'English'`, `'Other'`)                                    |
| `page_count`      | `float64` (nullable)  | `INTEGER`               | Количество страниц (`None` для eBook, Audiobook, Subscription)                |
| `genre`           | `object`              | `VARCHAR(50)`           | Жанр (из списка `GENRES`)                                                     |
| `format`          | `object`              | `VARCHAR(20)`           | Формат: `'eBook'`, `'Paperback'`, `'Hardcover'`, `'Audiobook'`, `'Subscription'` |
| `price_rub`       | `float64`             | `NUMERIC(10,2)`         | Розничная цена в рублях                                                       |
| `cost_rub`        | `float64`             | `NUMERIC(10,2)`         | Себестоимость (доля от цены по `MARGIN_RATIOS`)                               |
| `avg_rating`      | `float64`             | `NUMERIC(3,1)`          | Средний рейтинг (3.2 … 5.0)                                                   |
| `review_count`    | `int64`               | `INTEGER`               | Количество отзывов                                                            |
| `stock_initial`   | `float64` (nullable)  | `INTEGER`               | Начальный остаток для физических книг, `None` для остальных                   |
| `is_physical`     | `bool`                | `BOOLEAN`               | Производный признак: `True` для `Paperback`, `Hardcover`                      |

#### 2.3. `fact_plan` (Плановые показатели)

Планы формируются помесячно для комбинаций `genre × format × year × month`.

| Поле                 | Тип Pandas    | Предполагаемый тип в БД | Описание                                                |
|----------------------|---------------|-------------------------|---------------------------------------------------------|
| `plan_id`            | `int64`       | `INTEGER` / `SERIAL` PK | Уникальный идентификатор плана                          |
| `genre`              | `object`      | `VARCHAR(50)`           | Жанр                                                    |
| `format`             | `object`      | `VARCHAR(20)`           | Формат (только 4 книжных, без подписок)                 |
| `year`               | `int64`       | `INTEGER`               | Год                                                     |
| `month`              | `int64`       | `INTEGER`               | Месяц                                                   |
| `plan_qty`           | `int64`       | `INTEGER`               | Плановое количество проданных единиц (net)              |
| `plan_amount`        | `float64`     | `NUMERIC(12,2)`         | Плановая сумма продаж (net) в рублях                    |
| `plan_margin_target` | `float64`     | `NUMERIC(5,4)`          | Целевая маржинальность (доля валовой прибыли от выручки)|

**Особенности:**
- Для 2024 года план строится на основе фактических данных с мультипликатором 0.9–1.1.
- Для 2025 и 2026 – по данным аналогичного месяца предыдущего года с мультипликатором 0.92–1.20.
- Если фактов нет, план может быть нулевым.

#### 2.4. `fact_sales` (Факты продаж)

Генерируется построчно для каждого товара в заказе. Затем разбивается на две части: `sales_history` (до 15.04.2026) и `sales_update` (ровно на 15.04.2026).

| Поле                 | Тип Pandas | Предполагаемый тип в БД | Описание                                                         |
|----------------------|------------|-------------------------|------------------------------------------------------------------|
| `sales_id`           | `int64`    | `INTEGER` / `SERIAL` PK | Уникальный идентификатор строки продажи                          |
| `order_id`           | `int64`    | `INTEGER`               | Идентификатор заказа (в одном заказе может быть несколько позиций)|
| `product_id`         | `int64`    | `INTEGER`               | Ссылка на `dim_product.product_id`                               |
| `date`               | `date`     | `DATE`                  | Дата продажи                                                     |
| `sales_qty`          | `int64`    | `INTEGER`               | Количество проданных единиц                                      |
| `return_qty`         | `int64`    | `INTEGER`               | Количество возвращённых единиц                                   |
| `unit_price`         | `float64`  | `NUMERIC(10,2)`         | Цена за единицу с учётом скидки (если есть)                      |
| `sales_amount`       | `float64`  | `NUMERIC(12,2)`         | `sales_qty × unit_price`                                         |
| `return_amount`      | `float64`  | `NUMERIC(12,2)`         | `return_qty × unit_price`                                        |
| `discount_percent`   | `int64`    | `INTEGER`               | Процент скидки (0, если без промо)                               |
| `is_promo`           | `int64`    | `INTEGER`               | Признак применения промокода (0/1)                               |
| `promo_code`         | `object`   | `VARCHAR(20)`           | Применённый промокод или `None`                                  |
| `channel`            | `object`   | `VARCHAR(20)`           | Канал продаж (`'Web'`, `'Mobile App'`, `'API Partner'`)          |
| `lost_sales_qty`     | `int64`    | `INTEGER`               | Потерянные продажи из-за дефицита остатка (рассчитывается в постобработке)|

**Особенности:**
- Моделируется дефицит физических книг: если в какой-то день продажи превышают доступный остаток, часть `sales_qty` переносится в `lost_sales_qty`, а `sales_amount` пересчитывается.
- Пополнение запасов происходит периодически (раз в 30–90 дней) случайными партиями.
- Возвраты возможны только для физических книг (`Paperback`, `Hardcover`) с вероятностью 2%.
- Для подписок промо не предусмотрено (кроме намеренной аномалии №13).

#### 2.5. `fact_inventory` (Ежедневные остатки)

Только для физических форматов (`Paperback`, `Hardcover`).  
Разбивается на `inventory_history` и `inventory_update` аналогично продажам.

| Поле                | Тип Pandas | Предполагаемый тип в БД | Описание                                        |
|---------------------|------------|-------------------------|-------------------------------------------------|
| `inventory_id`      | `int64`    | `INTEGER` / `SERIAL` PK | Уникальный идентификатор записи                 |
| `product_id`        | `int64`    | `INTEGER`               | Ссылка на `dim_product.product_id`              |
| `date`              | `date`     | `DATE`                  | Дата                                            |
| `opening_stock`     | `int64`    | `INTEGER`               | Остаток на начало дня                           |
| `sold_qty`          | `int64`    | `INTEGER`               | Продано за день (net = продажи – возвраты)      |
| `replenishment_qty` | `int64`    | `INTEGER`               | Пополнение в этот день (0, если не было)        |
| `closing_stock`     | `int64`    | `INTEGER`               | Остаток на конец дня                            |
| `is_low_stock`      | `int64`    | `INTEGER`               | Признак низкого остатка (0/1, порог = 15)       |

### 3. Заложенные аномалии (режим `anomalies_enabled=True`)

Аномалии вносятся **после генерации «чистых» данных** и только при утвердительном ответе пользователя. Их цель – имитировать типичные проблемы качества данных.

#### 3.1. `dim_product`

| №  | Описание аномалии                                                       | Доля затронутых строк | Механизм реализации                                                                  |
|----|-------------------------------------------------------------------------|-----------------------|--------------------------------------------------------------------------------------|
| 1  | `author = NULL`                                                         | 1%                    | Замена на `np.nan`                                                                   |
| 2  | `publisher = NULL`                                                      | 1%                    | Замена на `np.nan`                                                                   |
| 3  | `page_count = NULL` у бумажных книг                                     | 2% от физических      | Для `Paperback`/`Hardcover` выставляется `np.nan`                                     |
| 4  | `isbn = 'INVALID_ISBN'`                                                 | 1%                    | Замена корректного ISBN на строку `'INVALID_ISBN'`                                    |
| 5  | `price_rub < 0`                                                         | 0.1%                  | Умножение цены на –1                                                                 |
| 6  | `cost_rub < 0`                                                          | 0.1%                  | Умножение себестоимости на –1                                                        |
| 7  | `stock_initial` задан для цифровых форматов (`eBook`/`Audiobook`)       | 1% от цифровых        | Присвоение случайного остатка (1–1000) там, где должно быть `None`                    |
| 23 | Пустая строка в одном из полей: `author`, `publisher`, `genre`          | 0.2%                  | Замена значения на `''` (пустая строка, не `NULL`)                                    |
| 8  | Дублирование строк                                                      | 0.3%                  | Копирование случайных строк с присвоением нового `product_id` (дубликаты данных)      |

#### 3.2. `fact_sales`

| №  | Описание аномалии                                                       | Доля затронутых строк | Механизм реализации                                                                                   |
|----|-------------------------------------------------------------------------|-----------------------|-------------------------------------------------------------------------------------------------------|
| 11 | Искажение `sales_amount`                                                | 0.2%                  | Умножение `sales_amount` на случайный коэффициент 0.5–1.5 (нарушение равенства `qty × price`)         |
| 12 | `return_qty > sales_qty` для физических книг                            | 0.5% от физических    | Установка `return_qty = sales_qty + 1` и пересчёт `return_amount`                                      |
| 13 | Промо-скидка 20% для подписок                                           | 0.5% от подписок      | Для строк с подпиской `is_promo = 1`, `discount_percent = 20` (нарушение бизнес-правила)              |
| 9  | Удаление 5% заказов целиком                                             | 5% от `order_id`      | Случайные `order_id` полностью удаляются из таблицы                                                    |
| 10 | Удаление всех записей за 1% случайных дат                               | ~1% дат               | Полное удаление строк, относящихся к выбранным датам                                                   |
| 14 | Дублирование строк с новым `sales_id`                                   | 0.5%                  | Копирование случайных строк с новым уникальным `sales_id`                                              |

#### 3.3. `fact_plan`

|----|---------------------------------|------------|-----------------------------------------|
| 17 | `plan_qty < 0`                  | 0.5%       | Замена на отрицательное значение        |
| 18 | `plan_amount < 0`               | 0.5%       | Замена на отрицательное значение        |

#### 3.4. `fact_inventory`

| №  | Описание аномалии               | Доля строк | Механизм реализации                                   |
|----|---------------------------------|------------|-------------------------------------------------------|
| 19 | `closing_stock = -5`            | 0.2%       | Принудительная установка отрицательного остатка       |

### 4. Разбиение на history и update

Функция `split_and_save_fact` делит таблицы фактов по дате **2026-04-15**:
- **history** – все записи **до** этой даты (строго `< 2026-04-15`).
- **update** – записи **за** эту дату (ровно `2026-04-15`).

**Почему это важно:**  
Такой подход имитирует сценарий инкрементальной загрузки:  
- `*_history` – основная историческая часть, загружается один раз.
- `*_update` – ежедневные изменения, которые будут добавляться в БД при последующих обновлениях.

В реальном проекте вы можете:
- Загрузить `history` файлы в хранилище как базовый срез.
- Настроить ежедневную загрузку `update` файлов (или потоковую репликацию изменений).

### 5. Автоматические проверки в скрипте

После генерации выполняются структурные проверки (ассерты), которые могут «мягко» пропускать некоторые нарушения, если включён режим аномалий:

| Проверка                                            | Ожидаемое поведение в чистом режиме                     | Поведение при аномалиях                     |
|-----------------------------------------------------|---------------------------------------------------------|---------------------------------------------|
| Отсутствие висячих `product_id`                     | Все `product_id` из фактов есть в справочнике           | Может быть нарушено (из-за удалённых строк) |
| `closing_stock ≥ 0`                                 | Нет отрицательных остатков                              | Аномалия №19 создаёт -5                     |
| Цифровые продукты отсутствуют в `fact_inventory`    | Должны отсутствовать                                    | Не нарушается                               |
| Даты фактов в пределах `dim_date`                   | Должны быть в границах 2024-01-01 … 2026-04-15          | Не нарушается                               |
| Полнота комбинаций в `fact_plan`                    | Может не быть некоторых комбинаций (допустимо)          | То же                                       |
| `return_qty ≤ sales_qty`                            | Всегда выполняется                                      | Аномалия №12 создаёт нарушения              |
| Нет возвратов у цифровых продуктов                  | Всегда выполняется                                      | Не нарушается                               |
| `sales_amount = sales_qty × unit_price` (с допуском) | Выполняется с точностью до копеек                       | Аномалия №11 искажает часть строк           |

### 6. Рекомендации по использованию данных

1. **Обработка аномалий** – при разработке ETL-процессов предусмотрите валидацию и очистку данных:
	 - Отрицательные цены и остатки должны либо отбрасываться, либо помечаться как ошибочные.
	- Пустые строки преобразуйте в `NULL`.
	- Дубликаты удаляйте или агрегируйте.
	- Нарушения бизнес-правил (например, возврат > продаж) логируйте и исправляйте.
2. **Инкрементальная загрузка** – `*_update` файлы содержат данные только за одну дату. В реальном проекте вы будете получать подобные файлы ежедневно и добавлять их в соответствующие таблицы фактов.
3. **Моделирование дефицита** – поле `lost_sales_qty` содержит оценку упущенных продаж. Это ценная метрика для анализа спроса и управления запасами.
