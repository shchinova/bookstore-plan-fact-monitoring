# Правила проверки качества данных

> Документ описывает правила валидации и очистки данных, импортируемых из 1С. Правила применяются при первичной загрузке (`load_history.py`) и при каждом ежедневном обновлении (`daily_update.py`). При изменении правил сначала обновляется этот документ, затем — код (`scripts/validate.py`).

## Общие правила

### Файловые и технические проверки

|Код|Правило|Действие|
|---|---|---|
|G-01|Разделитель столбцов — запятая (`,`). Строки с иным разделителем (например, `;`) считаются некорректными.|Отклонить строку, записать в отчёт|
|G-02|Кодировка файла — UTF-8. Строки с нечитаемыми символами (артефакты CP1251) считаются повреждёнными.|Попытаться восстановить; если не удалось — отклонить строку, записать в отчёт|
|G-03|Пустая строка ( , `''`, `""`) в текстовом поле приравнивается к `NULL`.|Заменить на `NULL`, записать в отчёт|
|G-04|Лишние пробелы в начале и конце строкового значения.|Обрезать (`strip()`), записать количество исправлений в отчёт|
|G-05|Полностью пустые строки (все поля `NULL` или пустые).|Удалить, записать в отчёт|

## `dim_date` — Календарь

Справочник дат на период проекта. Не обновляется ежедневно; пополняется раз в год. Правила:

|Код|Поле|Правило|Действие|
|---|---|---|---|
|D-01|`date`|Формат даты — `YYYY-MM-DD`. Любое отклонение (например, `15.04.2026`, `Apr 15 2026`) считается ошибкой.|Попытаться привести к стандарту; если не удалось — отклонить строку|
|D-02|`date`|Значение должно быть валидной календарной датой (не `2024-13-01`, не `2024-02-30`).|Отклонить строку|
|D-03|`date`|Дубликаты по полю `date` недопустимы.|Удалить дубликат, оставить первую запись, записать в отчёт|
|D-04|`year`, `month`, `quarter`|Должны соответствовать значению `date`. Например, для `2024-03-15`: `year=2024`, `month=3`, `quarter=1`.|Пересчитать из `date`, записать расхождения в отчёт|
|D-05|`day_of_week`|Должен соответствовать `date` (0 = понедельник … 6 = воскресенье).|Пересчитать из `date`, записать расхождения в отчёт|

## `dim_product` — Справочник продуктов

Мастер-данные по ~952 продуктам. Обновляется редко: добавление новых книг, изменение цен, снятие с продажи. Правила:

|Код|Поле|Правило|Действие|
|---|---|---|---|
|P-01|`product_id`|Уникальный идентификатор, `NOT NULL`. Дубликаты по `product_id` недопустимы.|Отклонить дублирующую запись, записать в отчёт|
|P-02|`product_id`|Тип — целое число, значение > 0.|Отклонить строку|
|P-03|`title`|`NOT NULL`, непустая строка.|Отклонить строку|
|P-04|`author`|Ожидается непустая строка. `NULL` допустим только для подписок (`format = 'Subscription'`).|Записать в отчёт как предупреждение|
|P-05|`publisher`|Ожидается непустая строка. `NULL` — предупреждение.|Записать в отчёт как предупреждение|
|P-06|`format`|Допустимые значения: `eBook`, `Paperback`, `Hardcover`, `Audiobook`, `Subscription`.|Отклонить строку|
|P-07|`isbn`|Для книг: валидный ISBN-13 (13 цифр, контрольная сумма по стандарту). Для подписок (`format = 'Subscription'`): `NULL`.|Записать в отчёт как предупреждение (не блокирует загрузку)|
|P-08|`price_rub`|`NOT NULL`, числовое, > 0. Отрицательные и нулевые значения недопустимы.|Отклонить строку|
|P-09|`cost_rub`|`NOT NULL`, числовое, > 0. Отрицательные и нулевые значения недопустимы.|Отклонить строку|
|P-10|`cost_rub`|Себестоимость не должна превышать цену (`cost_rub ≤ price_rub`).|Записать в отчёт как предупреждение|
|P-11|`page_count`|Для `Paperback` и `Hardcover`: ожидается целое число > 0. `NULL` — предупреждение. Для остальных форматов: ожидается `NULL`.|Записать в отчёт как предупреждение|
|P-12|`stock_initial`|Для `eBook` и `Audiobook`: должен быть `NULL` (цифровые товары не имеют физического остатка). Ненулевое значение — ошибка логики.|Обнулить (`NULL`), записать в отчёт|
|P-13|`avg_rating`|Числовое, диапазон 0.0–5.0 включительно.|Отклонить строку|
|P-14|`review_count`|Целое, ≥ 0.|Отклонить строку|
|P-15|—|Дубликат записи: совпадение по всем полям кроме `product_id`. Признак задвоения в 1С.|Записать в отчёт как предупреждение; в БД загружать только первую запись|

## `fact_plan` — Планы

Плановые показатели по продуктам и периодам. Обновляется раз в период (месяц или квартал). Правила:

|Код|Поле|Правило|Действие|
|---|---|---|---|
|PL-01|`product_id`|Должен существовать в `dim_product`. Висящий внешний ключ недопустим.|Отклонить строку|
|PL-02|`date`|Формат `YYYY-MM-DD`, валидная дата.|Попытаться привести; если не удалось — отклонить|
|PL-03|`plan_qty`|Числовое, ≥ 0. Отрицательные значения недопустимы.|Отклонить строку|
|PL-04|`plan_amount`|Числовое, ≥ 0. Отрицательные значения недопустимы.|Отклонить строку|
|PL-05|`plan_qty`, `plan_amount`|Если `plan_qty = 0`, то `plan_amount` должен быть 0, и наоборот.|Записать в отчёт как предупреждение|
|PL-06|`(product_id, date)`|Комбинация должна быть уникальной. Дубликаты плана на одну дату по одному продукту недопустимы.|Удалить дубликат, оставить первую запись, записать в отчёт|

## `fact_sales` — Продажи

Транзакционные данные о продажах. Обновляются ежедневно — `INSERT` строк за прошедший день. Правила:

|Код|Поле|Правило|Действие|
|---|---|---|---|
|S-01|`sales_id`|Уникальный идентификатор строки, `NOT NULL`. Дубликаты по `sales_id` недопустимы.|Удалить дубликат, записать в отчёт|
|S-02|`order_id`|`NOT NULL`. Все строки одного заказа должны иметь одинаковый `order_id`.|Записать в отчёт строки с `NULL`|
|S-03|`(order_id, product_id)`|Комбинация должна быть уникальной в рамках одного заказа. Дубликат позиции — признак задвоения.|Удалить дубликат, записать в отчёт|
|S-04|`product_id`|Должен существовать в `dim_product`.|Отклонить строку|
|S-05|`date`|Формат `YYYY-MM-DD`, валидная дата.|Попытаться привести; если не удалось — отклонить|
|S-06|`date`|Дата должна быть в пределах периода проекта: не раньше `2024-01-01` и не позже текущей даты.|Отклонить строку|
|S-07|`date`|Проверка полноты: если за какой-либо день в периоде нет ни одной строки продаж — это потерянный день.|Записать в отчёт как предупреждение (не блокирует загрузку)|
|S-08|`qty`|Целое, > 0.|Отклонить строку|
|S-09|`sales_amount`|Числовое, > 0.|Отклонить строку|
|S-10|`sales_amount`|Ожидаемая сумма: `qty × price_rub × (1 − discount_percent / 100)`. Расхождение более 1% — ошибка.|Записать в отчёт как предупреждение|
|S-11|`return_qty`|Целое, ≥ 0. Для цифровых товаров (`eBook`, `Audiobook`): ожидается 0 или `NULL`.|Записать ненулевые значения для цифровых товаров в отчёт|
|S-12|`return_qty`|Не должен превышать `qty` в той же строке.|Записать в отчёт как предупреждение|
|S-13|`return_amount`|Числовое, ≥ 0. Если `return_qty = 0`, то `return_amount` должен быть 0.|Записать расхождения в отчёт|
|S-14|`discount_percent`|Числовое, диапазон 0–100 включительно.|Отклонить строку|
|S-15|`is_promo`|Булево (0 или 1). Если `is_promo = 1`, то `discount_percent > 0` и `promo_code NOT NULL`.|Записать нарушения в отчёт как предупреждение|
|S-16|`is_promo`|Если `is_promo = 0`, то `discount_percent` должен быть 0.|Записать нарушения в отчёт как предупреждение|

## `fact_inventory` — Остатки

Остатки физических книг (Paperback, Hardcover) на конец каждого дня. Обновляются ежедневно. Правила:

|Код|Поле|Правило|Действие|
|---|---|---|---|
|I-01|`product_id`|Должен существовать в `dim_product`.|Отклонить строку|
|I-02|`product_id`|Только физические книги: `format IN ('Paperback', 'Hardcover')`. Записи для eBook, Audiobook, Subscription недопустимы.|Отклонить строку|
|I-03|`date`|Формат `YYYY-MM-DD`, валидная дата.|Попытаться привести; если не удалось — отклонить|
|I-04|`date`|Дата в пределах периода проекта: не раньше `2024-01-01` и не позже текущей даты.|Отклонить строку|
|I-05|`(product_id, date)`|Комбинация должна быть уникальной: один остаток на продукт на дату.|Удалить дубликат, записать в отчёт|
|I-06|`closing_stock`|Целое, ≥ 0. Отрицательные остатки недопустимы.|Отклонить строку|
|I-07|`is_low_stock`|Булево (0 или 1). Должен соответствовать условию: `is_low_stock = 1` если `closing_stock ≤ reorder_point`, иначе 0.|Пересчитать, записать расхождения в отчёт|

## Формат отчёта о качестве данных

Отчёт сохраняется в `reports/` в формате Markdown.  
Имя файла: `quality_report_YYYY-MM-DD_HH-MM.md`  
При ежедневном обновлении каждый запуск создаёт новый файл.

### Структура отчёта

```
# Отчёт о качестве данных
Дата запуска: YYYY-MM-DD HH:MM  
Скрипт: load_history.py / daily_update.py

## Сводка

|----------------|----------------|-----------|------------|----------------|-----------|
| dim_date       | ...            | ...       | ...        | ...            | ...       |
| dim_product    | ...            | ...       | ...        | ...            | ...       |
| fact_plan      | ...            | ...       | ...        | ...            | ...       |
| fact_sales     | ...            | ...       | ...        | ...            | ...       |
| fact_inventory | ...            | ...       | ...        | ...            | ...       |

## Детали по таблицам

### dim_product
- [P-08] Отрицательная цена: 1 строка отклонена (product_id: 412)
- [P-15] Дубликат записи: 3 строки (product_id: 801, 802, 803) — загружена только первая

- ...

```

## Приоритеты действий

| Приоритет                | Действие                                 | Когда применяется                                                             |
| ------------------------ | ---------------------------------------- | ----------------------------------------------------------------------------- |
| 🔴 Отклонить строку      | Строка не загружается в БД               | Критическая ошибка: нарушение типа, недопустимое значение, висящий FK         |
| 🟡 Исправить и загрузить | Значение исправляется автоматически      | Ошибка формата даты (если восстановима), пустая строка → NULL, лишние пробелы |
| 🔵 Предупреждение        | Строка загружается, аномалия фиксируется | Логическое несоответствие, отсутствие необязательного поля                    |
